# AGENTS.md

## 项目概览

EasyXL 是一个运行在 IXL 页面上的 Tampermonkey 用户脚本项目。当前仓库采用单脚本架构，主入口文件为 `easyxl.js`。脚本会在页面右下角注入悬浮面板，提取当前题目的 HTML，调用用户所配置的 AI 供应商接口进行解析和求解，并将最终答案显示在结果区域。

本文件面向 AI Agent，目标是让任何接手本项目的 Agent 都能快速理解当前实现方式、修改边界、开发约定和验证要求，在进行功能开发、错误修复、界面调整或文档维护时保持一致性与可维护性。

## 仓库结构

- `easyxl.js`
  - 项目唯一主脚本
  - 包含元数据、配置管理、UI 构建、题目提取、AI 请求适配、答案渲染、交互逻辑
- `README.md`
  - 英文说明文档
- `README-cn.md`
  - 中文说明文档
- `bookmarklet.txt`
  - 辅助性历史产物，不是当前主安装入口
- `LICENSE.md`
  - 许可证文件

当前仓库没有 `src`、`tests`、`package.json`、打包脚本、CI 配置或多模块目录。默认不要假设这是一个需要构建流程的 Web 工程。

## 项目架构

### 总体架构

项目为单文件、顺序式组织的浏览器端脚本，代码大致分为以下几个层次：

1. Userscript 元数据区
2. 常量与供应商配置定义
3. 本地配置加载、默认值合并、旧配置迁移
4. UI 元素创建与样式设置
5. Markdown 与数学公式渲染
6. Prompt 构建与题目 HTML 抽取
7. 不同 AI 供应商的请求适配层
8. 统一求解流程控制
9. 事件绑定、拖拽、快捷键、设置面板开关

### 运行流程

1. 用户在 IXL 页面加载脚本
2. 脚本读取 `localStorage` 中的统一配置
3. 页面右下角渲染 EasyXL 主悬浮窗
4. 用户点击齿轮进入设置面板，选择供应商并填写 API 信息
5. 用户点击“解析并求解”
6. 脚本从题目区域选择器中提取 `section` 的 `outerHTML`
7. 脚本构造 system prompt 与 user prompt
8. 按供应商分派请求
9. 模型返回完整解题文本
10. 脚本提取 `<answer>...</answer>` 中的最终答案
11. 将最终答案渲染到结果框，支持 Markdown 与 KaTeX 数学公式

### 当前关键设计原则

- 单入口：主逻辑集中在 `easyxl.js`
- 最小依赖：仅依赖 Tampermonkey API 与 CDN 加载的渲染库
- 向后兼容：保留旧独立脚本配置迁移逻辑
- 本地配置：所有用户设置均存储在浏览器 `localStorage`
- 原生实现：UI 使用原生 DOM 与内联样式，不引入前端框架

## 核心模块说明

### 1. Userscript 元数据

职责：
- 声明脚本名称、匹配站点、权限、远程依赖与可访问域名

修改约定：
- 新增供应商时，必须同步补充 `@connect`
- 若新增外部资源，必须确认当前项目确实需要，并尽量保持最少依赖

### 2. 供应商配置表

`PROVIDERS` 是统一供应商配置中心，包含：

- `label`
- `kind`
- `defaultModel`
- `defaultBaseUrl`
- `models`
- `apiKeyPlaceholder`
- `baseUrlPlaceholder`
- `notesPlaceholder`

修改约定：
- 新增供应商时，优先在此处完成默认配置定义
- 不要将供应商逻辑散落在多个互不关联的位置

### 3. 配置加载与迁移

统一配置键：

- `easyxl_unified_settings`

结构约定：

```json
{
  "selectedProvider": "openai",
  "providers": {
    "openai": {
      "apiKey": "",
      "model": "gpt-4o",
      "baseUrl": "https://api.openai.com/v1/chat/completions",
      "notes": ""
    }
  }
}
```

职责：
- 创建默认配置
- 合并不完整配置
- 兼容旧版本独立脚本配置
- 保存用户当前选择与供应商设置

修改约定：
- 新增字段时必须提供默认值
- 修改存储结构时必须考虑旧配置兼容
- 默认不要更换存储介质，除非用户明确要求

### 4. 主 UI 与设置面板

主界面职责：
- 展示当前供应商标识
- 提供用户注释输入
- 提供解析按钮
- 展示结果输出

设置面板职责：
- 选择供应商
- 配置 API Key
- 配置模型
- 配置基础 URL
- 展示错误提示

关闭机制要求：
- 右上角 X 按钮关闭
- ESC 键关闭
- 点击遮罩层关闭

修改约定：
- 新增交互时优先复用已有 UI 风格
- 保持液态玻璃风格与简洁布局
- 不要引入复杂动画或明显增加资源占用

### 5. 题目提取与 Prompt 构建

题目来源：
- 直接提取 IXL 页面中题目区域 section 的 `outerHTML`

当前选择器：
- `section.ixl-practice-crate`
- `section.question-and-submission-view`
- `section.question-view`

Prompt 约定：
- 模型要分析 HTML 还原题目
- 模型要输出解题步骤
- 模型必须将最终答案放入 `<answer>...</answer>`

修改约定：
- 不要轻易改变题目提取策略
- 不要移除 `<answer>` 协议，除非同步修改解析逻辑并得到明确需求支持

### 6. 请求适配层

当前支持供应商：
- OpenAI
- Anthropic
- Google
- DeepSeek
- Kouri

实现方式：
- OpenAI 兼容类接口共用一套请求格式
- Google 使用 `generateContent`
- Anthropic 使用 `messages`
- DeepSeek 包含 404 fallback 逻辑

修改约定：
- 新增供应商时优先复用 `requestByProvider` 分派模式
- 不要把请求逻辑直接写进按钮点击事件中
- 错误信息应尽量提取接口返回内容并反馈给用户

### 7. 结果渲染层

职责：
- 提取 `<answer>` 标签内的最终答案
- 渲染 Markdown
- 渲染数学公式
- 渲染错误信息

修改约定：
- 普通结果走 Markdown 与数学渲染
- 错误提示保持醒目、可读
- 不要直接把未经处理的 HTML 注入错误区

### 8. 页面交互

包含：
- 拖拽悬浮窗
- Ctrl 单击显隐
- 设置面板的打开与关闭

修改约定：
- 保持 Ctrl 显隐逻辑不影响组合键行为
- 不要让拖拽与按钮交互互相冲突

## 开发规范

### 代码组织规范

- 维持单文件的顺序式结构
- 优先在现有函数附近修改，而不是随意插入到文件任意位置
- 新逻辑应尽量放入语义清晰的辅助函数中
- 若修改涉及多个层面，优先遵循以下顺序：
  - 配置定义
  - 数据处理
  - 请求层
  - UI 层
  - 事件绑定

### 命名规范

- 常量使用全大写加下划线
- 变量与函数使用 camelCase
- 函数名尽量体现行为，例如：
  - `loadSettings`
  - `saveSettings`
  - `validateConfig`
  - `renderSettingsPanel`
  - `requestByProvider`

### 风格规范

- 使用 `const` 和 `let`
- 保持分号
- 保持 4 空格缩进
- 延续原生 DOM 构建风格
- 延续内联样式方式

### 修改策略

- 优先做最小必要改动
- 不为“看起来更工程化”而引入构建系统、框架或额外目录
- 除非用户明确要求，否则不要进行大规模重构
- 修改用户可见行为时，同步更新 `README.md` 和 `README-cn.md`

## 项目特定编码约定

以下约定是本项目的核心规则，AI Agent 修改时必须优先遵守：

1. 主入口固定为 `easyxl.js`
2. 配置统一存储在 `easyxl_unified_settings`
3. 旧配置迁移逻辑必须保留
4. 题目默认发送 HTML，而不是人工提取后的纯文本
5. AI 返回结果必须继续使用 `<answer>...</answer>` 进行最终答案提取
6. 网络请求默认继续使用 `GM_xmlhttpRequest`
7. UI 默认继续使用原生 DOM 和内联样式
8. 新供应商接入时，必须同步更新：
   - `PROVIDERS`
   - 设置面板表单
   - 配置校验
   - 请求分发
   - README 中英文说明
9. 错误提示默认双通路输出：
   - 结果区显示错误
   - 设置面板显示配置引导
10. 不要把 API Key 输出到控制台、页面或日志中

## 常见问题与解决方案

### 1. 页面没有出现 EasyXL 窗口

排查方向：
- 是否运行在 `ixl.com` 页面
- Tampermonkey 是否启用
- 脚本是否成功保存
- 浏览器控制台是否有报错

### 2. 提示找不到题目 HTML

排查方向：
- 当前页面是否真的是 IXL 题目页
- IXL 页面结构是否发生变化
- 题目选择器是否需要补充

处理建议：
- 检查现有三个 section 选择器
- 若页面结构已变，最小化修改选择器列表

### 3. 请求失败

排查方向：
- API Key 是否正确
- 模型名是否正确
- 基础 URL 是否正确
- 对应域名是否已写入 userscript `@connect`

处理建议：
- 保持“结果区报错 + 设置面板引导”双提示
- 优先展示接口返回的错误正文

### 4. 返回了步骤但没有提取最终答案

原因：
- 模型未按要求输出 `<answer>...</answer>`

处理建议：
- 优先修正 prompt
- 必要时增强提取兼容性，但不要无故移除 `<answer>` 协议

### 5. 数学公式显示异常

排查方向：
- `marked` 是否正常加载
- `KaTeX` 是否正常加载
- 返回文本中的数学格式是否异常

### 6. 配置丢失

原因：
- 浏览器清理了 `localStorage`
- 用户切换了浏览器配置文件

处理建议：
- 确认 `easyxl_unified_settings` 是否仍存在
- 不要假设配置是永久持久的

## AI Agent 修改指南

### 适合直接修改的场景

- 新增供应商
- 调整设置面板字段
- 优化错误提示
- 更新 Prompt
- 修复题目提取失败
- 优化结果渲染
- 更新 README

### 修改前必须确认的事实

- 当前是单脚本项目，不是多模块项目
- README 中英文都需要保持一致
- 用户可能已经在浏览器中保存了真实配置，不能破坏兼容性

### 推荐工作顺序

1. 阅读 `easyxl.js`
2. 确定改动属于哪个层次
3. 尽量在现有函数和结构内做最小修改
4. 修改后进行语法检查
5. 同步更新文档
6. 总结影响范围与验证结果

### 禁止事项

- 不要泄露或打印 API Key
- 不要默认删除迁移逻辑
- 不要在没有明确需求的情况下引入框架或打包工具
- 不要把当前单脚本架构擅自改成复杂多文件架构
- 不要只更新中文或只更新英文文档

## 文档维护要求

以下变化发生时，必须评估是否需要同步更新 AGENTS.md：

- 仓库结构变化
- 主入口变化
- 配置结构变化
- 供应商列表变化
- 请求协议变化
- 测试方式变化
- 安装方式变化
- 关键开发规范变化

AGENTS.md 的目标不是营销，而是帮助 Agent 正确做事。因此应保持：

- 事实准确
- 结构清晰
- 面向维护
- 与当前代码一致

## 总结

对 AI Agent 来说，EasyXL 的核心理解可以概括为一句话：

这是一个单文件 Tampermonkey 用户脚本项目，默认采用最小改动、向后兼容、原生 DOM、统一本地配置和多供应商请求分派的方式进行维护；任何功能改动都应优先保护题目 HTML 提取链路、`<answer>` 协议、设置面板交互和中英文文档一致性。
